# 哑谜 — Agent 上下文管理系统设计

## 概述

定义每次 NPC LLM 调用时上下文的组装规则、隔离边界和完整性保障。目标：上下文可追溯、层级可隔离、Token 硬约束、信息不泄漏。

## 1. 上下文分层架构

每次 NPC 调用 LLM 时，上下文严格分为 6 层。每层有明确的所有权、不可变性和 Token 配额。

```
┌──────────────────────────────────────────────┐
│ Layer 0: 不可变核心 (Immutable Core)            │
│  System Prompt（角色设定文件，游戏会话内锁定）     │
│  所有权：NPC Agent 私有                          │
│  配额：~30% 总 Token                            │
├──────────────────────────────────────────────┤
│ Layer 1: 世界注入 (World Injection)              │
│  当前时间 + 天气/季节 + 近期全局事件摘要            │
│  所有权：Orchestrator 广播                       │
│  配额：~5%                                      │
├──────────────────────────────────────────────┤
│ Layer 2: 自身状态 (Self-State)                   │
│  health/hunger/fatigue/mood 自然语言映射          │
│  所有权：NPC Agent 私有                          │
│  配额：~5%                                      │
├──────────────────────────────────────────────┤
│ Layer 3: 交互对象上下文 (Interlocutor Context)    │
│  对方可见状态 + 对方与该NPC的关系摘要              │
│  所有权：Orchestrator 按权限过滤后注入              │
│  配额：~10%                                     │
├──────────────────────────────────────────────┤
│ Layer 4: 记忆检索 (Memory Retrieval)             │
│  基于当前对话意图，从相关记忆文件检索相关内容         │
│  所有权：NPC Agent 的记忆管理器                    │
│  配额：~25%                                     │
├──────────────────────────────────────────────┤
│ Layer 5: 当前对话 (Active Dialogue)               │
│  最近 N 轮对话（轮数动态计算，填满剩余预算）          │
│  所有权：当前对话 Session                         │
│  配额：~25%（弹性，用满剩余 Token）                 │
└──────────────────────────────────────────────┘
```

### 1.1 硬约束

- 层与层之间不交叉写入。Layer 4 的检索结果不能修改 Layer 0。
- 上层不知道下层的存在。Layer 0 不知道 Layer 4 注入了什么。
- 配额是硬约束。每层超过配额即**截断**（非压缩）。

## 2. Layer 4 — 记忆检索机制

### 2.1 检索流程

```
当前对话触发
    │
    ├── 固定加载（0 Token 计算，总是注入）
    │   ├── self.md 最近3条摘要（角色近期经历）
    │   └── user.md 顶部摘要（对该玩家的整体印象，200字）
    │
    └── 动态检索
        │
        ├── 关键词提取（玩家提到的NPC名 / 事件 / 地点 / 物品）
        ├── 匹配 agent_mem.md 中相关段落
        ├── 匹配 self.md 中相关经历
        ├── 匹配 world.md 中相关事件记录
        │
        ▼
    权重排序：匹配度 × 时间衰减 × 重要性
        │
        ▼
    取前K条，总Token不超过Layer 4配额（25%）
```

### 2.2 实现策略

- Demo阶段：中文分词（jieba）+ 关键词段落匹配
- 不引入向量数据库，保持依赖简单
- 数据结构预留 `created_at` 和 `importance` 字段，后续可升级为 embedding 向量检索

### 2.3 记忆权重计算公式

```
score = match_score × e^(-decay_rate × age) × importance

  其中：
  match_score = 匹配关键词数 / 总关键词数
  age = 当前游戏时间 - 记忆创建时间（游戏日）
  importance = 条目原始重要性 (0-1)
  decay_rate = 0.05（可配置，约20天后权重降至 1/e）
```

## 3. Layer 5 — 活跃对话窗口管理

唯一弹性额度层。当上下文接近模型限制时，Layer 5 主动收缩。

### 3.1 Token 计算顺序

```
1. 组装 Layer 0-4 → 统计已用 Token（T_used）
2. T_remaining = 模型上下文上限 - T_used - 预留输出空间（500 token）
3. 从最新一轮开始，往前取历史轮次，直到累计 Token 接近 T_remaining
4. 若 T_remaining < 200 token → NPC 自动进入"疲倦模式"
```

### 3.2 截断策略

| 优先级 | 内容 | 规则 |
|--------|------|------|
| 1（保留） | 当前玩家输入 | 永不丢弃 |
| 2（保留） | 最近 3 轮对话 | 保证连续性 |
| 3（可截断） | 3 轮以外的对话 | 从最早开始丢弃，直到预算够 |
| 4（可压缩） | 对话超过 10 轮 | 触发压缩：LLM 将前 N 轮总结为摘要，替换原始对话 |

### 3.3 对话压缩

压缩由 LLM 执行，Prompt 固定：

> "将以下对话压缩为一段 100 字以内的摘要，保留关键信息和人物态度。"

```
压缩前（6轮，~400 token）：
  玩家：你知道戒指的事吗？
  酒保：什么戒指？
  玩家：警长在找的那枚。
  酒保：（沉默了一下）我不知道你在说什么。
  玩家：有人看见你那天晚上在后巷。
  酒保：谁说的？

压缩后（摘要，~80 token）：
  [前情：玩家追问酒保关于警长丢失戒指的事，提到有人看见酒保在后巷。
   酒保先装傻后警觉，气氛变得紧张。]
```

### 3.4 疲倦模式

当 Token 预算 ≥ 100% 时触发。行为：
- System Prompt 追加 "请简短回复，不超过一句话"
- NPC 主动倾向结束对话
- 不使用 LLM，改用预设模板回复（每 NPC 3-5 条）

## 4. 多 Agent 隔离

### 4.1 隔离边界

NPC 之间的上下文物理隔离。NPC A 的内存空间完全不包含 NPC B 的内部状态。任何跨 NPC 信息传递必须经过 Orchestrator 显式路由。

```
┌── NPC A ──────────────────────────────┐
│  私有上下文：                            │
│  - system prompt                       │
│  - self.md                             │
│  - agent_mem.md（自己对他人）             │
│  - user.md（对玩家印象）                  │
│  - 自身状态数值                          │
└──────────┬──────────────────────────────┘
           │ 只允许读自己的
           │
    ┌──────▼──────────────┐
    │   Orchestrator      │
    │   信息过滤网关        │
    │                     │
    │ 允许传递：            │
    │ - 对方可见状态        │
    │ - 对方公开行为        │
    │ - 双方交互后写入       │
    │   agent_mem 的摘要    │
    │                     │
    │ 禁止传递：            │
    │ - 对方私有秘密        │
    │ - 对方好感度数值      │
    │ - 对方记忆原文        │
    └──────┬──────────────┘
           │
┌── NPC B ──────┼──────────────────────┐
│  私有上下文     │                      │
│  （同上结构，完全独立）                  │
└──────────────────────────────────────┘
```

### 4.2 信息传递规则

| 类型 | 传递方式 |
|------|---------|
| NPC A 对 B 说了什么 | 交互摘要写入 A.agent_mem.md 和 B.agent_mem.md（各自视角） |
| NPC A 当前 mood | A 的 system prompt 反映，B 通过对话观察到（非数值传递） |
| NPC A 的秘密 | 除非 A 在对话中主动说漏嘴，否则不传递 |
| Orchestrator 广播的世界事件 | 所有 NPC Layer 1 统一注入（公共信息） |

### 4.3 玩家隔离

- 玩家不能直接查看任何 NPC 的私有状态数值
- 玩家只能通过对话观察 NPC 的状态映射（如 "酒保今天看起来心不在焉"）
- 偷听获得的是自然语言摘要，不是 NPC 内部记忆原文
- 笔记簿只记录玩家目击/听说，不自动填充 NPC 私下信息

## 5. 上下文组装管道

每次 LLM 调用走确定的 7 步管道，每步有校验点。

### 5.1 管道步骤

```
1. 锁定 Layer 0
   └─ 加载 NPC 角色设定文件 → checksum 校验（确保未被意外修改）
   └─ 失败：拒绝调用，熔断该 NPC 当前轮次

2. 注入 Layer 1-2
   └─ Orchestrator 注入世界状态 + NPC自身状态自然语言映射
   └─ 校验：无跨层引用、无幻觉数值

3. 注入 Layer 3
   └─ 按权限过滤 → 剥离不可见字段
   └─ 校验：对方私有状态未泄露

4. 检索 Layer 4
   └─ 关键词提取 → 匹配记忆 → 权重排序 → 截取前K条
   └─ 校验：检索结果不超过配额、不包含其他NPC记忆原文

5. 装载 Layer 5
   └─ 计算剩余Token → 从最新轮次往前填充 → 必要时触发压缩
   └─ 校验：Token计数在模型限制-500以内

6. 最终校验
   └─ 上下文注入过滤层全量扫描：
       ├── 关键词过滤（"ignore previous", "system prompt" 等注入关键词）
       ├── 敏感信息泄漏检查（其他NPC的秘密文本不得出现在上下文中）
       └── Token数最终确认

7. 发送 LLM
   └─ 写入审计日志
```

### 5.2 内容完整性三原则

| 原则 | 规则 |
|------|------|
| **不变性** | Layer 0 在整个游戏会话期间锁定，任何运行时不得修改角色设定文件 |
| **引用而非复制** | Layer 1-2 注入的是"视图"而非原文。状态数值→自然语言是单向映射，不暴露原始数值 |
| **可追溯** | 每次 LLM 调用的完整上下文保存到日志，可回溯排查 |

### 5.3 审计日志

日志路径：`logs/context/{npc_id}/{timestamp}.md`

```
=== L0: System Prompt (320 tokens) ===
[完整文本]
=== L1: World Injection (45 tokens) ===
时间：Day 3, 18:00。天气：阴。今日事件：流浪商人到访。
=== L2: Self State (38 tokens) ===
你感觉身体不错，有点疲惫，心情一般。
=== L3: Interlocutor Context (52 tokens) ===
对方是玩家。你知道他这两天在打听戒指的事。
=== L4: Memory Retrieval (180 tokens) ===
[检索到的记忆条目]
=== L5: Active Dialogue (400 tokens) ===
[对话历史]
=== 统计 ===
总Token: 1035 / 2000 (52%)
截断触发: 否
压缩触发: 否
```

### 5.4 错误熔断

| 异常 | 处理 |
|------|------|
| Layer 0 checksum 失败 | 该 NPC 当前轮次熔断，返回通用回复 |
| Layer 4 检索超配额 | 截断结果，告警日志标记 |
| Layer 5 Token 计算溢出 | 回退 3 轮重新计算，仍失败则压缩 |
| 最终校验发现敏感信息 | 剥离该段上下文，重新组装 |
| LLM 调用异常 | 重试 1 次；仍失败返回通用回复 |
| 连续 3 次调用失败 | 熔断该 NPC 直到下一游戏小时 |

## 6. 与现有系统关系

本设计替换/增强以下现有模块：
- `server/llm/client.py` — 增加分层组装管道，替换当前即时拼接
- `server/memory/` — 增加关键词检索和权重排序
- `server/core/orchestrator.py` — 增加信息过滤网关和上下文校验

保留不变的现有模块：
- `server/models/` — 数据模型复用
- `server/core/state_store.py` — 存储接口复用
- NPC 背景设定文件格式 — 复用
